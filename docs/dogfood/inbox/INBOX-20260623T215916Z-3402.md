## Suspected fix
Invariants validated/derived from the on2it-pr-council reference workflow (see
docs/design/loop-architecture.md §3, §6):
- The test-gate verdict is the execution anchor for the election quorum: an
  APPROVE/winner may not pass without it present and passing.
- Graders/runners operate in a harness-isolated worktree, NEVER the operator's live
  checkout; isolation is enforced by the harness, not by a prompt instruction.
- Pin the build/test to the candidate's exact commit SHA so every lens judges the
  same revision (the reference build lens did this; its read-only peers did not).
- Emit a structured verdict, not just PASS/FAIL: { pass, checks_run[] (command +
  key result), blocking_issues[] } so the result is auditable downstream.
- Don't trust supplied scan/evidence files — re-run the check in the worktree.
